### CAPITAL MODEL LEVEL

**C1: Scale-out trades treated as separate pool positions (ROOT CAUSE of -$9,513)**
- File: `utils/capital_model.py` line 174
- `for row_idx, row in tdf.iterrows()` iterates ALL trade records including scale-outs
- A position with 2 scale-outs creates 3 Trade384 records (SCALE_1, SCALE_2 or SL/TP)
- All 3 share the same `entry_bar` -- capital model treats each as needing $500 margin
- One $500 position consumes $1,500 in pool margin
- With max_positions=4 per coin, 2 coins, max_scaleouts=2: up to 24 phantom positions = $12,000 > $10k pool
- Fix: Group trades by position (coin + entry_bar), create ONE event per position

**C2: Double margin deduction in available calculation**
- File: `utils/capital_model.py` lines 222-223
- `pool_balance -= margin_per_pos` at entry (line 227) deducts margin from pool
- `margin_in_use = len(active) * margin` counts active margins
- `available = pool_balance - margin_in_use` subtracts margin AGAIN
- Fix: Separate `balance` and `margin_used` tracking. `available = balance - margin_used`

## Engine Trade Flow (Verified by Audit)

### How positions map to Trade384 records:
- Fresh A/B/C/D entry: market order, taker commission, creates PositionSlot384
- ADD/RE entry: limit order fills in Step 4, maker commission
- Scale-out: fires in Step 3, creates Trade384 with scale_idx=1,2,...
  - do_scale_out reduces slot.notional by close_notional
  - If is_final (scale_count >= max_scaleouts): slot set to None, no further SL/TP
- SL/TP exit: fires in Step 1, creates Trade384 with scale_idx=0
  - Uses remaining notional after any prior scale-outs
  - close_at passes full remaining entry_commission (bug: doesn't account for already-allocated shares)
- END close: forced at last bar, same path as SL/TP with exit_reason="END"

### Commission flow:
- Entry: `equity -= comm_entry`, stored in `slot.entry_commission`
- Exit: `equity += trade.pnl - comm_exit`
- Trade384.commission = entry_commission + exit_commission (both sides)
- Trade384.pnl = gross PnL (price movement * notional, no commission)
- trades_df.net_pnl = pnl - commission
- Rebate: settled daily at 17:00 UTC, added to equity, tracked in comm.total_rebate
- metrics.net_pnl = sum(trade.pnl - trade.commission) -- does NOT include rebate
- metrics.net_pnl_after_rebate = net_pnl + total_rebate

### equity_curve:
- Mark-to-market: equity_curve[i] = realized_equity + unrealized_pnl_from_open_positions
- At last bar: all positions force-closed, so equity_curve[-1] = pure realized

## Correct Pool Model (Exchange-Style)

```
balance = total_capital     (realized cash)
margin_used = 0             (locked capital)

Group trades by POSITION (coin + entry_bar):
  exit_bar = max(exit_bar) in group
  total_pnl = sum(net_pnl) in group
  -> ONE event per position

Process events sorted by entry bar:
  1. Close expired: exit_bar <= current_bar
     balance += total_pnl
     margin_used -= margin_per_pos
  2. available = balance - margin_used (NO double deduction)
  3. available >= margin_per_pos? ACCEPT, margin_used += margin_per_pos
  4. Else REJECT
```

Root cause: Session 3 fix added rebate as lump sum at end (`balance += accepted_total_rebate`). But during the simulation, the pool never received rebate income -- trades paid commission on open/close, pool balance only got net_pnl on close. Without daily rebate flowing back, gross commission drained the pool below margin threshold.

### Fix: Daily rebate settlement in pool simulation (Bug C3)

**Concept**: On real exchanges, rebate settles daily at 5pm UTC. The pool simulation must mirror this. Track `daily_comm_accumulated` from accepted positions. At each 5pm UTC boundary crossing, credit `daily_comm * rebate_pct` to `balance`.

**Implementation in `capital_model_v2.py`**:
- Function signature: `apply_capital_constraints(..., rebate_pct=0.70, settlement_hour=17)`
- Main event loop: before processing each bar, check if we crossed a 5pm UTC boundary. If yes and daily commission > 0, settle rebate to balance.
- Pool balance history loop: identical settlement logic per bar.
- Removed old lump-sum `balance += accepted_total_rebate` at end.
- Added PATCH 13 to dashboard: passes `rebate_pct=rebate_pct` from sidebar to function.

**Verification with `debug_pool_balance_v2.py`**:
- WITHOUT daily rebate: 195 rejections (10.5%), pool drains to $491, min available $4.57
- WITH daily rebate: 0 rejections, pool stays above $5,762, final pool $12,480
- 30 settlement events over 30 days
- Sanity check delta = $0.000000

### capital_model_v2.py -- Code Audit (537 lines)

**Functions (7)**:
1. `_grade_sort_key(grade_str)` -- Grade priority mapping (A=0, B=1, ..., R=6)
2. `_map_bar_to_master(bar_idx, coin_dt_index, master_dt)` -- Bar index translation
3. `_group_trades_into_positions(coin_results, master_dt)` -- Groups Trade384 records by (coin, entry_bar). Uses dict for O(1) lookup, then sorts by (entry_bar, -strength). Collapses scale-outs into one position with aggregated net_pnl/commission.
4. `_rebuild_metrics_from_df(tdf, eq_curve, orig_metrics, notional)` -- Rebuilds metrics after trade rejection. Scales volume/sides/rebate by commission ratio.
5. `apply_capital_constraints(...)` -- Main function. Exchange-model pool simulation with daily rebate settlement. Returns adjusted pf_data, coin_results, rejection log, capital_efficiency dict.
6. `_bar_info(bar_idx, label)` -- Helper for best/worst moments (nested in apply_capital_constraints).
7. `format_capital_summary(efficiency)` -- Display-ready string lines.

**Pool simulation flow**:
```
balance = total_capital, margin_used = 0
For each position event (sorted by entry_bar):
  1. Check 5pm UTC boundary -> settle daily rebate to balance
  2. Close expired: exit_bar <= bar -> balance += pnl, margin_used -= margin
  3. available = balance - margin_used
  4. Accept: margin_used += margin, track commission for rebate
  5. Reject: log rejection with balance/margin/available state
Final: settle remaining daily commission as rebate
```

**Pool balance history**: Per-bar step function built separately (second pass over bars). Same daily rebate settlement logic. Used for capital chart overlay and DD%/best/worst computation.

**Two rebased curves**:
- `rebased_portfolio_eq` = pool_balance_history (realized cash, step function) -- used for Net P&L, DD%, best/worst
- `rebased_chart_eq` = adjusted_portfolio_eq - engine_baseline + total_capital (smooth visual) -- used for equity chart
