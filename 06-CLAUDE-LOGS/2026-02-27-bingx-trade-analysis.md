### What the chart shows at 18:40 (5m, APT)
- Stoch 9: ~96-97 (still elevated, possibly just crossed below 75)
- Stoch 14: ~98 (above 70, going up)
- Stoch 40: ~98.82 (above 70, NEVER approached 80 line)
- Stoch 60: ~99.02 (above 75, NEVER approached 80 line)

**The SHORT A fired but there was no visual rotation. Stoch40 and Stoch60 were going UP near 100, not curling down.**

### Root cause — state_machine.py
The state machine only checks: "was Stoch40 ever > 70 during Stage 1?" — trivially true at 98. It does NOT check if Stoch40/60 are actively rotating downward. A proper quad rotation SHORT requires Stoch40 and Stoch60 to actually start moving DOWN from overbought (crossing below 80), not just being above 70 while still trending up.

### The LONG that should have been there
At 18:45 on the 5m chart: all 4 stochastics were in oversold territory. A LONG setup was visible but no LONG signal was generated by the bot. The bot was already IN the SHORT from 18:40 (wrong direction), so even if a LONG fired it would have been a simultaneous opposite position.

### 1m vs 5m confusion (important note)
User showed initial chart on 1m. Earlier analysis about ATR filter blocking a LONG at 18:45 was based on 1m chart data. On the 5m chart, ATR was 0.0040 (not 0.002x). The ATR filter did NOT block the APT SHORT — it passed with ratio 0.0040/0.96 = 0.0042 > 0.003.

### Grade B at 19:20
After the missed LONG at 18:45, a LONG B fired at 19:20 — after the big move had already run. Chasing entry. Confirms Grade B signals arrive late.

### Issues identified
1. **State machine missing Stage 2** — Stoch40/60 must actively rotate (cross below 80 for SHORT, above 20 for LONG) before signal can fire. Currently only checks if they were "seen" above threshold.
2. **Grade B produces chasing entries** — fires after the move, not at the start

4. **Trailing TP** — not implemented

## BingX VST API Oddities
1. Filled orders vanish from history within seconds (allOrders returns nothing)
2. STOP orders fill in wrong direction (above entry for LONG SL)
3. Both SL and TP pending orders disappear simultaneously on position close
4. VST infrastructure separate from live (Indonesian IPs, no CDN, datacenter-blocked)
5. Per-coin position limits undocumented and inconsistent
6. Mark price drifts 1.5% from signal price between generation and execution
7. 1m→5m EXIT_UNKNOWN ratio drops from ~100% to 6% (suggests rate limit/queue issue on VST)

## Features Not Implemented
- **Breakeven raise**: Not in production code. Only tested in lifecycle script (steps 8-9).
- **Trailing TP**: Not in production code. TP set once at entry, never modified.
- BingX API supports both (`TRAILING_STOP_MARKET`, `TRAILING_TP_SL` order types) but not wired into bot.
