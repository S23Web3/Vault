### ADD Definition (Critical Clarification)

ADD is not a stochastic signal type. It is a label applied by the execution engine when:
1. A Quad or Rotation signal fires
2. A same-direction position is already open
3. Capacity and cooldown conditions are met

ADD is never generated by the state machine. The current Pine C signal (2/4 stochs) is incorrect and must be removed from both Pine and Python.

## Architecture Findings

### Pine v3.8.2 — What It Currently Does

State machine generates three signal types:
- `long_signal` / `short_signal` — Quad (4/4, Cloud 3 bypassed)
- `long_signal_b` / `short_signal_b` — Rotation (3/4, Cloud 3 gated)
- `long_signal_c` / `short_signal_c` — C (2/4, Cloud 3 gated) ← WRONG, must be removed

C signals are suppressed in practice via `i_bOpenFresh = false` and `i_allowCTrades = false`, but the underlying logic is incorrect by design — not by parameterisation.

### Python Backtester — What It Currently Does

- `state_machine.py` generates `long_signal_c` / `short_signal_c` as DataFrame columns
- `backtester_v384.py` assigns grade C as a fresh entry fallback
- No `i_bOpenFresh` equivalent — C opens fresh positions regardless of existing position state
- Result: C trades open as independent entries, which is wrong

### Gap Summary

| Component | Current | Required |
|-----------|---------|----------|
| Pine state machine | Generates Quad + Rotation + C (2/4) | Generate Quad + Rotation only |
| Pine execution | C as degraded fresh entry | ADD = engine label when entry fires while positioned |
| Python state_machine.py | Generates `long_signal_c` column | Remove C generation entirely |
| Python backtester | Grade C assigned as fresh fallback | ADD label applied when same-direction position exists |
| Python config | `allow_c_trades = False` | Rename to `allow_add`, controls scale-in permission |
