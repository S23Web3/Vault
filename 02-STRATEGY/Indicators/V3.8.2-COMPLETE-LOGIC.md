## Overview

v3.8.2 replaces the fixed ATR SL/TP + breakeven system from v3.8 with a dynamic AVWAP-based 3-stage stop-loss management system. Positions are runners with no take profit — exits occur only when the trailing stop is hit.

**Core Concept:** Anchor AVWAP at each entry, use sigma bands for SL progression, allow pyramiding with independent position tracking.

---

## Entry System (Unchanged from v3.8)

### Quad Rotation Stochastic Signals

Four stochastic oscillators (all Raw K, smoothing=1):
- **9-3** (fastest): Entry trigger — enters oversold/overbought zone, then exits
- **14-3**: Confirmation #1
- **40-3**: Confirmation #2  
- **60-10**: Confirmation #3 (slowest)

### Signal Grades

- **A Trade (4/4):** 9-3 exits zone + all 3 others confirmed in zone during lookback → highest conviction
- **B Trade (3/4):** 9-3 exits zone + 2/3 others confirmed → medium conviction (optional, toggled by `i_allowBTrades`)
- **C Trade (2/4 + Cloud 3):** 9-3 + 14-3 confirmed + price position confirms Cloud 3 direction → lowest conviction (optional, toggled by `i_allowCTrades`)

### Two-Stage State Machine

- **Stage 0:** Waiting for 9-3 to enter oversold (<25) or overbought (>75) zone
- **Stage 1:** 9-3 in zone, tracking which other stochs also enter zone within `i_stageLookback` bars (default: 10)
- **Trigger:** When 9-3 exits zone → count confirmations → fire A, B, or C signal
- **Timeout:** If 9-3 stays in zone > `i_stageLookback` bars → reset to Stage 0

### Cloud 3 Directional Filter (ALWAYS ON)

Price position relative to Cloud 3 (EMA 34/50):
- **Price > Cloud 3 top:** Only LONG entries allowed
- **Price < Cloud 3 bottom:** Only SHORT entries allowed
- **Price inside Cloud 3:** Both directions allowed

### Cloud 2 Re-entry

When enabled (`i_cloud2Reentry`): if a signal (A/B/C) fired within `i_reentryLookback` bars and price crosses Cloud 2 (EMA 5/12) in the signal direction → re-enter. Only when flat.

### Entry Priority Chain

1. A signal (can flip: close opposite + open new)
2. A signal (fresh entry when flat)
3. B/C signal (fresh entry when flat, if `i_bOpenFresh` enabled)
4. Cloud 2 re-entry (when flat)
5. AVWAP add (limit order — see Section below)
6. AVWAP re-entry (limit order — see Section below)

### Cooldown

Minimum `i_cooldown` bars (default: 3) between entries. Prevents rapid-fire entries.

## Position Management (NEW in v3.8.2)

### Multi-Position Architecture

- **4 position slots** tracked via parallel arrays
- Each slot stores: entry price, entry bar, direction, SL stage, SL level, AVWAP accumulators, grade, ID
- Positions are independent — each has its own AVWAP anchor and SL stage
- Maximum `i_maxPositions` simultaneous positions (default: 4)
- `pyramiding=4` in strategy settings

### AVWAP Calculation (Per Position)

Each position anchors its own AVWAP from entry bar:

```
AVWAP = Σ(hlc3 × volume) / Σ(volume)
Variance = Σ(hlc3² × volume) / Σ(volume) - AVWAP²
Sigma = max(√Variance, sigmaFloor × ATR)
```

- **hlc3** = (high + low + close) / 3 — typical price
- **Sigma floor** (`i_sigmaFloor`, default: 0.5 × ATR) — prevents zero-width bands on first bars
- Bands: ±1σ and ±2σ from AVWAP center

### Three-Stage Stop Loss

**Stage 1: AVWAP ±2σ (initial)**
- LONG SL: AVWAP - 2σ (below entry)
- SHORT SL: AVWAP + 2σ (above entry)
- Risk = 2σ from center (but entry was at market, so actual risk depends on entry vs AVWAP)
- Dynamic: updates each bar as AVWAP evolves

**Stage 1 → Stage 2 Trigger:**
- Default (`opposite_2sigma`): Price reaches the opposite 2σ band (went in favor)
  - LONG: high ≥ AVWAP + 2σ
  - SHORT: low ≤ AVWAP - 2σ
- Alternative: After 5 or 10 bars (`bars_5`, `bars_10`)

**Stage 2: AVWAP ± ATR (tighter)**
- LONG SL: AVWAP - (ATR × `i_slAtrMult`)
- SHORT SL: AVWAP + (ATR × `i_slAtrMult`)
- Duration: `i_stage2Bars` bars (default: 5)
- Tighter than Stage 1 — locks in more profit

**Stage 2 → Stage 3 Trigger:**
- After `i_stage2Bars` bars in Stage 2

**Stage 3: Cloud 3 ± ATR (trailing)**
- LONG SL: Cloud 3 bottom - (ATR × `i_slAtrMult`)
- SHORT SL: Cloud 3 top + (ATR × `i_slAtrMult`)
- Trails with Cloud 3 — follows the macro trend
- Final stage, stays until stopped out

### Ratchet Guard

SL only moves in the favorable direction:
- LONG: SL can only increase (never moves down)
- SHORT: SL can only decrease (never moves up)

### Critical Fix: Execution Order

**v3.8 Bug:** `strategy.exit()` was called BEFORE stops were updated → positions could exit on stale SL levels
**v3.8.2 Fix:** Update stops FIRST (Section 13), THEN issue `strategy.exit()` with updated SL

Order each bar:
1. Accumulate AVWAP data
2. Compute AVWAP bands
3. Check stage transitions
4. Compute target SL
5. Apply ratchet guard
6. Issue `strategy.exit()` with final SL
7. Clean up closed positions
8. Process pending limits
9. Check new entries

## AVWAP Add System (NEW in v3.8.2)

### Concept

When price stretches to 2σ from a position's AVWAP, place a LIMIT order at 1σ to scale in. This adds at a mean-reversion level — buying dips in an uptrend, selling rallies in a downtrend.

### Mechanics

- **LONG add:** Price dips to AVWAP - 2σ → place LIMIT BUY at AVWAP - 1σ
- **SHORT add:** Price rises to AVWAP + 2σ → place LIMIT SELL at AVWAP + 1σ
- **Cancel:** If not filled after `i_cancelBars` bars (default: 3)
- **One at a time:** Only one pending limit order active
- **AVWAP age check:** Don't add on stale AVWAPs older than `i_maxAvwapAge` bars (default: 50)
- **Gated by:** Cloud 3 direction, cooldown, max positions

### Fill Detection

Each bar checks if the pending order ID appears in `strategy.opentrades`. If found:
- Allocate new position slot
- Initialize with fresh AVWAP from fill bar
- Clear pending state

## AVWAP Re-entry System (NEW in v3.8.2)

### Concept

After a stop-out, the AVWAP and sigma from the stopped position are "frozen." For `i_reentryWindow` bars (default: 5), if price revisits the 2σ band of the frozen AVWAP → place a LIMIT order at 1σ to re-enter.

### Mechanics

- On stop-out: Save `reentry_bar`, `reentry_dir`, `reentry_avwap`, `reentry_sigma`
- Window check each bar: if `bar_index - reentry_bar > i_reentryWindow` → expire
- **LONG re-entry:** Price dips to frozen AVWAP - 2σ → LIMIT BUY at frozen AVWAP - 1σ
- **SHORT re-entry:** Price rises to frozen AVWAP + 2σ → LIMIT SELL at frozen AVWAP + 1σ
- One attempt only — after placing limit, clear re-entry state
- If limit not filled → cancel after `i_cancelBars`

## Architecture Notes

### Why Arrays Instead of Single Variables

v3.8 tracked one position with scalar variables. v3.8.2 needs independent tracking per position because:
- Each position has its own AVWAP anchor (different entry bars)
- Each position is in a different SL stage
- Stop-out of one position shouldn't affect others
- Re-entry uses the stopped position's frozen AVWAP, not another position's

### Why Limit Orders for Adds

Market orders at 2σ would execute at potentially worse prices. Limit orders at 1σ ensure:
- Better fill price (mean-reversion entry)
- Natural fill: price must come back toward AVWAP to fill
- Risk/reward: entering at 1σ with SL at 2σ = 1σ risk

### Why No Take Profit

v3.8.2 is a runner strategy. The hypothesis:
- Trailing stop (especially Cloud 3 trail in Stage 3) captures more profit than fixed TP
- Fixed TP cuts winners short — especially in trending markets
- The 3-stage SL acts as dynamic profit protection
